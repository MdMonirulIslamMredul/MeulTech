## 4. Recommended Laravel Folder Architecture

Goal: Clean, testable code using service & repository layers, actions, Livewire components and helpers.

Example layout (inside `app/`):

- `app/Actions/` — single-responsibility action classes (CreateOrder, SendInvoice)
- `app/Services/` — domain services coordinating logic (CartService, InventoryService)
- `app/Repositories/` — repository interfaces & implementations (EloquentProductRepository)
- `app/Models/` — Eloquent models (Product, Category)
- `app/Policies/` — authorization policies
- `app/Http/Controllers/` — API and web controllers (thin; delegate to services/actions)
- `app/Http/Livewire/` — Livewire components for interactive UIs
- `app/Http/Resources/` — API Resources for mobile app / SPA
- `app/Observers/` — model observers for side effects (indexing, cache invalidation)
- `app/Jobs/` — queued jobs
- `app/Events/` & `app/Listeners/` — domain events
- `app/Exceptions/` — custom exceptions
- `app/Helpers/` — pure helper functions utilities
- `app/Console/Commands/` — artisan commands

Repository pattern:

- `app/Repositories/Contracts/ProductRepositoryInterface.php`
- `app/Repositories/Eloquent/ProductRepository.php`
  Bind interfaces to implementations in a service provider (e.g., `App\Providers\RepositoryServiceProvider`).

Service layer:

- Keep business logic here; controllers orchestrate request -> validate -> service -> response.

Actions:

- Use single-action invokable classes for atomic tasks. Example:
  - `app/Actions/Cart/AddItemToCart.php`
  - `app/Actions/Order/CreateOrderFromCart.php`

Livewire:

- Use Livewire for cart widget, product filters, product quick-view, checkout steps.
- Place components under `app/Http/Livewire/` and group by domain: `Shop`, `Account`, `Admin`.

Helpers:

- `app/Helpers/formatting.php` (money formatting), `media.php` (generate image variants), etc.

API readiness:

- Keep a dedicated `api` guard and separate controllers/resources in `app/Http/Controllers/Api/V1/`.

## 5. Scalable Database Planning

Use normalized tables with indexes for search-heavy fields. Use UUIDs or bigints depending on integration needs. For high-scale, use separate read replicas and a search index (Meilisearch). Example tables with key columns:

### Categories

- `id` (bigint)
- `parent_id` (nullable)
- `name`, `slug`, `description`
- `meta_title`, `meta_description`
- `position`, `is_active`
- timestamps

### Brands

- `id`, `name`, `slug`, `logo`, `description`

### Products

- `id`, `sku`, `type` (simple/variable/downloadable)
- `name`, `slug`, `short_description`, `description`
- `brand_id`, `category_id` (main), `status`
- `price`, `cost`, `special_price`, `special_price_start`, `special_price_end`
- `weight`, `dimensions`
- `tax_class_id`
- `is_featured`, `is_searchable`
- timestamps

### Product Specifications (attributes)

- `id`, `product_id`, `attribute_key`, `attribute_value`
- For global attributes, use `attributes` and `attribute_values` tables with pivot `product_attribute_value`.

### Product Variants

- `id`, `product_id`, `sku`, `price`, `compare_at_price`, `stock`, `barcode`, `attributes_json`
- `attributes_json` stores variant combination (e.g., {"color":"black","storage":"128GB"})

### Product Images / Media

- `id`, `model_type`, `model_id`, `collection`, `file_name`, `mime_type`, `size`, `meta`
- Use spatie/laravel-medialibrary schema or custom media table

### Orders

- `id`, `order_number` (human), `user_id` (nullable), `status`, `total`, `sub_total`, `shipping_cost`, `tax_amount`
- `billing_address_id`, `shipping_address_id`
- `payment_status`, `shipping_status`
- timestamps

### Order Items

- `id`, `order_id`, `product_id`, `variant_id` (nullable), `sku`, `name`, `qty`, `unit_price`, `total_price`, `tax_amount`

### Payments

- `id`, `order_id`, `gateway`, `gateway_transaction_id`, `amount`, `currency`, `status`, `meta` (json)

### Coupons

- `id`, `code`, `type` (fixed/percentage), `value`, `min_order_value`, `usage_limit`, `usage_per_customer`, `starts_at`, `ends_at`, `is_active`

### Reviews

- `id`, `product_id`, `user_id`, `rating`, `title`, `comment`, `status`, `approved_by`, `approved_at`

### Wishlists

- `id`, `user_id`, `name`, timestamps
- `wishlist_items`: `wishlist_id`, `product_id`, `variant_id`

### Compare System

- Lightweight per-user or session-based table or cache record storing product ids to compare

### Flash Sales / Promotions

- `id`, `name`, `starts_at`, `ends_at`, `status`, `rules` (json)
- `flash_sale_products` pivot: `flash_sale_id`, `product_id`, `discount`, `priority`

Indexes & scaling:

- Indexes on `slug`, `sku`, `status`, `brand_id`, `category_id` and searchable fields
- Use read replicas for analytics and heavy read routes
- Offload full-text & facet search to Meilisearch

## 9. Best Practices

### Scalability

- Use stateless web servers; store sessions in Redis
- Use queues for heavy tasks (emails, indexing, image processing)
- Use CDN for static assets and optimized image sizes
- Read replicas for reporting queries
- Use Meilisearch for product search and facets

### Security

- Use HTTPS everywhere and secure cookies
- Avoid storing sensitive card data; use tokenized gateway flows
- Rate limit critical endpoints and use CSRF protection
- Sanitize user inputs and validate thoroughly
- Keep dependencies up-to-date and use automated security scanning

### Performance Optimization

- Cache rendered fragments and API responses with Redis
- Use eager-loading to avoid N+1 queries
- Optimize images and serve WebP where possible
- Use queue workers for non-blocking tasks
- Use database indexes for frequent query fields

### Clean Code Architecture

- Thin controllers, fat services/actions
- Use interfaces for repositories for easier testing & swapping
- Keep contracts small and focused
- Unit tests for services, integration tests for core flows

- Keep styling via Tailwind utility classes and small, reusable component classes

## 14. Best Development Order to Avoid Future Problems

1. Define data models & migration contracts before UI
2. Implement repository contracts and bind them early
3. Implement background job & queue patterns before heavy features (indexing, image processing)
4. Add search integration early (Meili) so indexing hooks exist as models evolve
5. Build API resources in parallel with web controllers
6. Add tests as features are added: unit tests for services, feature tests for flows

## Appendix: Quick Local Setup Instructions

Run locally (once Composer & PHP environment available):

```bash
composer create-project --prefer-dist laravel/laravel Meultech "12.*"
cd Meultech
cp .env.example .env
composer require livewire/livewire filament/filament spatie/laravel-permission spatie/laravel-medialibrary meilisearch/laravel-scout
php artisan key:generate
php artisan migrate
php artisan serve
```

Notes:

- Replace `composer require` with the specific packages chosen and configure each package per docs.
- Configure Meilisearch and Redis services and set `SCOUT_DRIVER=meilisearch` in `.env`.
